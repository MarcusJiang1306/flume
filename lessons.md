# 重构经验与教训

## 问题与解决方案

### 1. 依赖注入与状态管理
- **问题**：useFlowEditor 作为服务容器，导致组件间依赖复杂
- **解决方案**：使用 Pinia 进行全局状态管理，替代服务容器模式
- **教训**：集中化的状态管理可以显著简化组件间的依赖关系

### 2. 类型定义问题
- **问题**：Ref 类型与普通类型的转换导致 TypeScript 错误
- **解决方案**：在需要 Ref 类型的地方使用 `ref()` 包装，在需要普通类型的地方使用 `.value`
- **教训**：明确区分 Ref 类型和普通类型，避免类型不匹配

### 3. 事件处理与类型转换
- **问题**：Vue Flow 事件返回的类型与应用内部类型不匹配
- **解决方案**：创建类型转换函数，将 GraphNode 和 GraphEdge 转换为内部类型
- **教训**：处理外部库事件时，需要注意类型转换

### 4. 方法调用问题
- **问题**：Pinia store 中方法调用失败，提示方法不存在
- **解决方案**：确保方法在 actions 中正确定义，避免使用动态属性访问
- **教训**：在 Pinia store 中定义方法时，确保方法名正确且可访问

### 5. 废弃 API 处理
- **问题**：测试文件仍然使用已废弃的 useFlowEditor
- **解决方案**：将 useFlowEditor 标记为废弃，引导使用新的 Pinia store
- **教训**：废弃旧 API 时，需要提供清晰的迁移指南

## 最佳实践

### 1. 状态管理
- 使用 Pinia 进行全局状态管理，集中管理应用状态
- 定义清晰的 state、getters 和 actions
- 避免在组件中直接管理状态，通过 store 操作状态

### 2. 类型定义
- 为所有状态和方法添加明确的 TypeScript 类型
- 避免使用 `any` 类型，尽可能使用具体类型
- 对于复杂类型，使用接口定义

### 3. 组件设计
- 组件职责单一，只负责渲染和用户交互
- 通过 props 接收数据，通过事件发送操作
- 避免组件间直接依赖，通过 store 通信

### 4. 代码组织
- 按功能模块组织代码，如 stores、composables、components
- 使用清晰的命名规范，提高代码可读性
- 保持代码简洁，避免冗余和重复

## 技术栈使用

### Vue 3 Composition API
- 使用 `setup` 语法糖，简化组件逻辑
- 利用 `computed`、`ref` 等响应式 API
- 合理使用 `inject`/`provide` 进行依赖注入

### Pinia
- 利用 Pinia 的响应式特性，实现状态的自动更新
- 使用 getters 计算衍生状态
- 通过 actions 处理复杂的业务逻辑

### TypeScript
- 严格使用 TypeScript 类型，提高代码质量
- 利用接口和类型别名，增强代码可读性
- 配置 TypeScript 编译选项，确保类型安全

## 总结

通过这次重构，我们成功将应用从服务容器模式迁移到 Pinia 状态管理模式，简化了组件间的依赖关系，提高了代码的可维护性和可扩展性。在重构过程中，我们遇到了一些类型和方法调用的问题，但通过仔细分析和解决，最终实现了一个更加清晰和高效的架构。

这次重构不仅解决了当前的问题，也为未来的功能扩展和代码维护奠定了良好的基础。
